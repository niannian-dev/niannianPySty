# 四区骨架优先的 GUI 重排计划

## Summary
把当前 `gui_app.py` 的窗口重排为“root 四行 + 主体两列”的骨架结构，先把布局体系理顺，再承载现有功能。

本轮目标不是删功能，而是：
- 根窗体只用 `grid`
- 固定 4 个一级 `Frame`：顶部、输入、主体、底部
- 主体区用 `grid` 拆成左右两栏
- 保留现有“输入提示 + 两层选择框 + 后台执行”能力，只是把它们挂到新的骨架里

## Key Changes
- 在 [gui_app.py](C:/Users/shan/Desktop/python/code/class4/mini_chatgpt_code/gui_app.py) 重构根窗体布局：
  - `root` 只负责四行 `grid`
  - 四个一级区域改成固定命名：`top_frame`、`input_frame`、`content_frame`、`status_frame`
  - 仅 `content_frame` 所在行设置可扩展权重，其余三行保持自然高度
- 重构顶部信息区：
  - 先保留“程序标题 + 当前工作目录 + 当前模型 + 当前请求模式”
  - 顶部区本身固定高度感，不承担可扩展空间
  - 内部可继续使用 `grid` 做信息项排列，避免和根窗体混乱
- 重构输入区为两行：
  - 第一行放说明文字/输入提示，并保留当前已有提示内容
  - 第二行放“输入框 + 按钮组”，按钮组单独放到子 `Frame` 中横向排列
  - 当前的“两层选择框”保留，并放入输入区内部的独立子区域，不挤占输入框主排布
- 重构主体区为左右两列：
  - `content_frame` 内用 `grid` 做左任务区、右结果区
  - 左右列都设置可拉伸；初版可做 1:1，或右侧略宽
  - 左侧固定为“标题 + Text”
  - 右侧固定为“标题 + Text”
  - 暂不引入复杂组件；如果有滚动条，继续保留在各自区域内部
- 重构底部状态区：
  - 只保留单一状态文本，不承载日志内容
  - 状态文案覆盖当前已出现的状态，如“空闲 / 处理中 / 写入等待确认 / 执行完成 / 执行失败”
- 保持现有控制层接口不变：
  - 不改 `AppController.process_input(...)` 的对外职责
  - 不改后台线程执行模型，只调整 GUI 如何承载和展示它

## Public Interfaces
- GUI 外观结构会变为明确的四区骨架，但用户已有操作方式保持不变：
  - 仍然通过输入框输入自然语言
  - 仍然通过执行按钮触发后台任务
  - 仍然在左侧看任务 JSON、右侧看执行结果
- 不修改 [controller.py](C:/Users/shan/Desktop/python/code/class4/mini_chatgpt_code/core/controller.py) 的调用协议
- 不修改工具层与任务 JSON 协议

## Test Plan
- 启动窗口后，根界面明确分为顶部、输入、主体、底部四块，且窗口放大时主要是主体区扩展
- 输入区保持两行结构，输入框和按钮不会因提示区或选择框而挤乱
- 主体区左右两栏都能随窗口拉伸，左侧任务区和右侧结果区都正常显示多行文本
- 底部状态区始终只显示单行状态，不出现大段结果文本
- 现有功能不回退：
  - 输入提示仍可见
  - 两层选择框仍可填充输入框
  - 点击执行后仍进入后台任务
  - 左侧仍显示任务 JSON，右侧仍显示执行结果或错误信息
- 根窗体和主要骨架容器不再混用 `pack`/`grid`；同一父容器内只使用一种布局管理器

## Assumptions
- 本轮以“骨架优先”为主，不删除当前已有功能，只重新挂载到新布局
- 主体区本轮先用 `grid` 两列实现，不直接切到 `PanedWindow`
- 左右两列初版按接近 1:1 处理即可；若需要右侧更宽，可在实现时轻微倾斜权重，但不引入额外交互复杂度
- 版本仍基于当前 [version.ver](C:/Users/shan/Desktop/python/code/class4/mini_chatgpt_code/version.ver) 的 `v0.0.3a` 继续演进
